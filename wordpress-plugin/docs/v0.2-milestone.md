# v0.2 Milestone

## Summary

v0.2 turns Crocoblock Site Factory from a working Real Estate demo into a clearer plugin beta foundation.

The plugin can generate the Real Estate vertical slice, create proof artifacts, auto-create the Request Viewing form when JetFormBuilder is available, and prove native Crocoblock filtering on an experimental page.

This milestone does not replace the stable frontend catalog filters yet. `/properties/` still uses the Render Adapter GET-filter implementation by design. The experimental `/properties-native/` page proves native JetSmartFilters rendering separately.

## What Is Now Automated

- Real Estate dashboard flow:
  - Editable prompt capture for run manifests
  - Explicit safe preset variables for selected hero/contact copy
  - Preview Plan
  - Generate Real Estate Demo
  - Validation proof
  - Advanced / Developer proof section
- Request Viewing flow:
  - Fallback Request Viewing card when JetFormBuilder is unavailable
  - Fallback treated as a valid beta state
  - Automatic JetFormBuilder Request Viewing form creation when JetFormBuilder is available
  - Single property CTA to `/contact/?factory_property={slug}`
- JetEngine Query Builder groundwork:
  - Query Builder object for Real Estate properties
  - Query ID: `factory_real_estate_properties`
  - Experimental native Query Builder object for native filtering
  - Native query ID: `native_list`
- Listing Adapter v2:
  - Property Card listing binds to the generated Query Builder row through numeric `_query_id`
- JetSmartFilters Adapter groundwork:
  - Generates filter definitions for:
    - Purpose
    - Property Type
    - District
  - Uses native taxonomy mode for generated filters
  - Uses Factory markers to avoid overwriting unrelated manual filters
- Native JetSmartFilters experimental proof page:
  - Generates `/properties-native/`
  - Renders three JetSmartFilters select blocks
  - Renders a JetEngine Listing Grid connected to `native_list`
  - Adds structural validation for query, filter, and page bindings
  - Exposes a secondary dashboard link for native proof checks
  - Leaves `/properties/` unchanged

## Current Architecture Flow

```text
Real Estate preset
  -> JetEngine CPT / meta / taxonomy adapters
  -> JetEngine Query Builder Adapter
     -> factory_real_estate_properties
     -> native_list experimental query
  -> Listing Adapter v2
     -> Property Card listing with numeric _query_id
  -> JetSmartFilters Adapter v1
     -> Purpose / Property Type / District native taxonomy filters
  -> Render Adapter
     -> Home, Properties, Contact pages
     -> stable GET filters on /properties/
     -> experimental native filters proof on /properties-native/
  -> Single Adapter
     -> polished property detail pages
  -> Content Adapter
     -> properties, meta, terms, images
  -> validation and run manifests
```

The Query Builder, listing binding, and JetSmartFilters definitions are groundwork for native Crocoblock filtering. `/properties-native/` proves the native path, but it is intentionally separate from the current stable frontend filter UI.

## Native JetSmartFilters Proof

Native JetSmartFilters rendering is proven on `/properties-native/`, not promoted to the main catalog yet.

The stable catalog remains:

- `/properties/`
- Render Adapter GET filters
- Existing beta-safe card grid and result count

The experimental native proof is:

- `/properties-native/`
- JetSmartFilters select controls for Purpose, Property Type, and District
- JetEngine Listing Grid
- Provider/query ID: `native_list`

The working native schema is intentionally strict:

- Query Builder query is a basic posts query.
- Query Builder custom ID is `native_list`.
- Listing Grid `_element_id` is `native_list`.
- Filter block `query_id` is `native_list`.
- SmartFilters use taxonomy mode:
  - `data-query-type="tax_query"`
  - `data-query-var="purpose"`
  - `data-query-var="property_type"`
  - `data-query-var="district"`
- SmartFilters do not use custom query variable / `_plain_query` mode.
- Query Builder `tax_query` remains empty because JetSmartFilters injects filtering at runtime.

## Intentional Fallbacks

The following fallbacks are expected and valid in v0.2:

- `/properties/` uses Render Adapter GET filters.
- `/properties-native/` is experimental proof and does not replace `/properties/`.
- Request Viewing fallback renders when JetFormBuilder is missing or no configured form is available.
- Missing optional JetSmartFilters support should not make a normal beta run fail.
- Prompt Testing v1 records prompt context in manifests.
- Prompt Testing v1 Phase 2 overlays explicit safe variables onto selected copy fields only.
- Prompt Testing v1 does not mutate schema, property content, filters, forms, queries, listings, media, or page topology.

## Not Implemented Yet

- Native JetSmartFilters rendering promoted to `/properties/`
- Native AJAX filter UI
- Price range and bedrooms filters as native JetSmartFilters definitions
- JetSmartFilters replacement for the existing GET-filter frontend
- JetSmartFilters validation against promoted frontend behavior
- Map listing
- Agent profiles
- JetFormBuilder schema editor
- Generic prompt-to-site generation
- Free-prose prompt parsing into variables
- Arbitrary prompt-driven blueprint mutation
- SaaS provisioning or external deployment

## Demo Checklist

Before a demo:

1. Confirm Kava is active.
2. Confirm JetEngine is active.
3. Optional: confirm JetFormBuilder is active if demonstrating native form creation.
4. Optional: confirm JetSmartFilters is active if demonstrating generated filter definitions.
5. Open **Site Factory** in WordPress admin.
6. Click **Preview Plan**.
7. Click **Generate Real Estate Demo**.
8. Confirm validation proof is OK.
9. Open `/`.
10. Open `/properties/`.
11. Test GET filters:
    - Purpose
    - Property Type
    - District
    - Bedrooms
    - Price min / max
12. Open a single property page.
13. Click **Contact agency**.
14. Confirm `/contact/?factory_property={slug}` renders the Request Viewing context.
15. If JetSmartFilters is active, confirm generated definitions exist in WordPress:
    - Purpose
    - Property Type
    - District
16. Optional native proof: open `/properties-native/`.
17. On `/properties-native/`, confirm Purpose / Property Type / District filters update the Listing Grid.
